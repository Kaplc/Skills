# Impl-Flow — 一句话实现流程

**核心目标**：用最短的文字说清楚怎么做，让用户一眼看懂实现思路。

---

## 默认输出格式

收到需求后，**直接输出一句话流程**，根据场景分两种格式：

### 实现类（实现XX / XX怎么做）

```
[入口] → [步骤1] → [步骤2] → ... → [结果]
```

**原则**：
- 箭头数量控制在 3～6 个步骤
- 每个节点用动词短语说清楚"做什么"
- 把关键技术名内嵌进去（如 JWT、Redis、WebSocket）
- 不写废话，不解释为什么，不加前缀"实现思路是："

**示例**：

用户：用户登录怎么实现？
输出：前端收集账号密码 → POST 到后端接口 → 验证账号密码 → 生成 JWT token → 返回客户端存入 localStorage → 后续请求 Header 携带 token 鉴权

---

用户：扫码支付怎么做？
输出：前端调支付接口生成订单 → 后端请求支付宝/微信生成二维码链接 → 前端渲染二维码 → 用户扫码确认 → 支付平台回调后端 → 后端更新订单状态 → 前端轮询或 WebSocket 通知支付成功

---

### 修改类（修改XX）

格式：`[文件/位置] 的 [具体字段/方法] → 改为/调整 [目标值/逻辑] → 使 [效果/影响]`

**原则**：
- 说清楚**在哪改**（文件名或模块）
- 说清楚**改什么**（字段、方法、参数）
- 说清楚**有什么作用**（改完之后系统行为怎么变）
- 如果涉及多处联动修改，用箭头串联

**示例**：

用户：修改用户头像上传的大小限制
输出：`upload.js` 中 `limits.fileSize` 控制上传大小上限 → 改为目标值（如 `5 * 1024 * 1024`）→ 使超限请求返回 413 而非静默失败

---

用户：修改接口的超时时间
输出：`axios.js` 中 `timeout` 字段设置全局超时毫秒数 → 改为目标值（如 `10000`）→ 使慢接口不再被提前断开，超时后统一走 error 拦截器处理

---

如果用户说"详细说"、"展开一下"、"每步说清楚"，则输出**分步骤列表**：

```
1. [步骤名称]
   做什么：具体操作
   涉及：技术/工具/接口

2. [步骤名称]
   ...
```

保持每步简洁，不超过 3 行，不写代码，不写完整设计文档。

---

## 注意事项

- 需求模糊时，基于最常见的实现方式作答，不需要反问
- **"修改XX"** 场景：严格按修改类格式，说清楚文件位置、改什么字段/方法、改完有什么效果
- **"实现XX"** 场景：从最上游入口开始，串联核心链路，到结果/输出结尾
- 不要主动输出详细版，等用户要
- 不要输出完整技术方案文档，那是 `skl-idea-documentor` 的职责
