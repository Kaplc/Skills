# Idea-Documentor - 需求实现文档生成器

**核心目标**: 将用户模糊的需求转化为结构化的实现思路文档

**输出要求**: 严格按照 TEMPLATE.md 模板格式生成文档，所有占位符必须替换为实际内容

---

## 工作流程概览

```
用户需求 → Stage 1 需求理解 → Stage 2 参考收集 → Stage 3 思路构建 → Stage 4 生成文档
```

---

## Stage 1: 需求理解与分析

**输入**: 用户原始需求描述
**输出**: 结构化的需求要素（见下方格式）

### 处理步骤

1. **澄清需求** - 理解需求的背景和使用场景
2. **提取要素** - 识别核心功能、关键行为、数据流转、参与者、触发条件、预期结果
3. **识别边界** - 确定系统边界和约束条件（非功能需求）

### ⚠️ 重要：目标不明确时必须询问

**执行顺序**：先自行分析 → 发现不明确目标 → 立即询问用户

**必须明确的关键目标**：

| 目标项 | 说明 | 不明确时 |
|--------|------|----------|
| 核心功能 | 用户想实现什么 | ❌ 询问：具体是哪个功能 |
| 触发条件 | 什么操作/事件触发 | ❌ 询问：何时触发 |
| 预期结果 | 期望达到什么效果 | ❌ 询问：期望什么效果 |
| 参与者 | 谁参与这个功能 | ⚠️ 可自行推断，建议询问确认 |
| 边界情况 | 需要处理哪些异常 | ⚠️ 可自行分析，建议询问确认 |

**自行分析原则**：
- ✅ 能从上下文、项目已有代码推断的，自行推断
- ✅ 能从用户描述中合理假设的，先假设并在输出中注明
- ❌ 核心目标（功能/触发/结果）不明确时，**不得自行假设**，必须询问

**询问格式**：
```
分析您的需求后，有以下几点不明确，需要确认：

❓ {问题1}: {具体问题}
   - 例如: 触发条件: "是游戏开始时自动播放，还是需要用户手动点击播放？"

❓ {问题2}: {具体问题}
```

### 输出格式

```
══════════════════════════════════════════════════════════════
                       需求理解分析结果
══════════════════════════════════════════════════════════════

📋 原始需求:
{用户原始需求原文}

📌 需求概述:
{一句话描述核心功能}

🎯 关键行为:
   - {行为1}: {描述}
   - {行为2}: {描述}

👥 参与者/角色:
   - {角色1}: {职责}
   - {角色2}: {职责}

⚠️ 约束条件:
   - {约束1}
   - {约束2}

══════════════════════════════════════════════════════════════
```

## Stage 2: 参考实现收集

**输入**: Stage 1 的需求要素
**输出**: 参考实现列表（表格形式）

### 处理步骤

1. **搜索项目内实现** - 使用 `search_content` / `search_file` 搜索相关类
2. **提取核心方法** - 找到关键函数/方法，标注其签名和用途
3. **评估相关度** - 高/中/低三档
4. **添加文档跳转** - 提供可点击的路径链接或 `@see` 引用
5. **必要时搜索外部** - 使用 `web_search` 查找最佳实践

### 项目内部参考要求

对于每个项目内部参考，**必须**提供：

| 字段 | 说明 | 示例 |
|------|------|------|
| 类型 | "项目内部" 固定值 | 项目内部 |
| 名称 | 类/模块名称 | PlayerComponent |
| 路径 | **Markdown 可跳转链接** | `[PlayerComponent.cs](file:///e:/Project/.../PlayerComponent.cs)` |
| 相关度 | 高⭐⭐⭐ / 中⭐⭐ / 低⭐ | ⭐⭐⭐ |
| 核心方法 | 方法签名 + 一句话用途 | `Play(): 开始播放`<br>`Pause(): 暂停播放` |
| 文档跳转 | **Markdown 可跳转链接** | `[@see PlayerComponent.Play()](file:///...#L15)` |

### 文档跳转链接格式规范

**路径链接格式**：
```
[文件名.cs](file:///{绝对路径})
[文件名.cs](file:///{绝对路径}#L行号)           ← 单行跳转
[文件名.cs](file:///{绝对路径}#L开始-L结束)     ← 多行跳转
```

**示例**：
```markdown
[AttackHandler.cs](file:///e:/UnityProject/Framework/Assets/Scripts/GameDomain/Entity/Component/Battle/Attack/AttackHandler.cs#L20-L35)
```

**约束**：
- ✅ 必须使用 `file:///` 协议
- ✅ 必须使用**绝对路径**（从盘符开始）
- ✅ 行号用 `#L` 前缀
- ✅ 多个方法用 `<br>` 分隔

### 输出格式

```

══════════════════════════════════════════════════════════════

## Stage 3: 实现思路构建

**输入**: Stage 1-2 的分析结果
**输出**: 实现方案、逻辑链、优缺点分析

### 3.1 设计实现方案

必须包含以下字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| 方案名称 | 清晰易记的名称 | "模块化组件方案" |
| 核心思路 | 一句话描述核心思想 | "通过事件驱动解耦各模块" |
| 实现步骤 | 序号列表，每步描述具体做什么 | 1. 定义事件接口 2. 实现模块... |
| 模块划分 | 模块名: 职责 | ConfigModule: 配置文件管理 |
| 数据结构 | 核心数据结构设计 | Dictionary<string, Component> |
| 关键算法 | 核心算法描述 | 使用对象池管理组件生命周期 |

### 3.2 构建逻辑链

描述从触发到完成的完整流程：

```
入口 → {处理1} → {决策点} → {处理2} → 结束
```

### 3.3 优缺点分析

必须包含：
- **优点**: 至少 3 条，每条说明为什么是优点
- **缺点**: 至少 2 条，每条说明 + 缓解措施
- **适用场景**: 至少 2 个
- **不适用场景**: 至少 1 个

### 输出格式

```
══════════════════════════════════════════════════════════════
                       实现思路构建结果
══════════════════════════════════════════════════════════════

📝 方案名称: {方案名称}
💡 核心思路: {核心思路}

📌 实现步骤:
   1. {步骤1}
   2. {步骤2}
   3. {步骤3}

🏗️ 模块划分:
   - {模块1}: {职责}
   - {模块2}: {职责}

══════════════════════════════════════════════════════════════
```

## Stage 4: 生成实现思路文档

**输入**: Stage 1-3 的所有分析结果
**输出**: 完整的 Markdown 文档

### 处理步骤

1. 读取 TEMPLATE.md
2. 替换所有占位符为实际内容
3. 保存到 `{workspace}/docs/需求文档_{日期}.md`

### 约束

- **必须替换所有占位符** - 不允许存在 `{xxx}` 格式的未填充内容
- **表格必须完整** - 不能留空单元格，用 "-" 表示
- **代码块需标注语言** - 如 ` ```csharp `

## 常见问题处理

### Q: 用户需求太模糊怎么办？
A: 主动提问澄清关键点：触发条件、预期结果、边界情况

### Q: 找不到参考实现怎么办？
A: 扩大搜索范围，或使用 web_search 搜索外部方案

### Q: 优缺点分析不全面怎么办？
A: 从以下维度检查：
- 可行性（能实现吗）
- 复杂度（实现难度）
- 性能（效率如何）
- 可维护性（以后好改吗）
- 可扩展性（以后好加功能吗）
